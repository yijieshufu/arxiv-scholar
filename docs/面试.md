## 二、技术栈全景：用了什么，解决什么问题

> 面试官最爱问「你为什么选这个技术而不是那个」  
> 每一行都要能回答「解决了什么痛点」

### 2.1 PDF 解析层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **Docling** | 布局树分析：区分标题/段落/表格，原生 `export_to_html()` 保留 colspan/rowspan | pyMuPDF 没布局树；pdfplumber 网格提取丢合并单元格；Camelot 安装复杂 |
| **pdfplumber** | Docling 降级回退——没 GPU 或 Docling 崩溃时纯文本提取 | 比 PyPDF2 准（有文字定位），比 Docling 轻 |
| **PyPDF2** | 最后兜底——最轻量纯文本提取 | 准度差（无定位），只保证有东西 |
| **物理拆批 (PyPDF2 split)** | Docling C++ preprocess 阶段 OOM 崩溃，Python try/except 抓不住 | page_range 参数不能阻止预处理，必须拆成独立物理文件 |

### 2.2 切片层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **SectionChunker** | 固定窗口切分把 Method 尾和 Experiment 首切进同一块 → 上下文污染 | 按章节边界切（Abstract/Intro/Method/Exp/Conclusion），每块语义完整 |
| **表格原子块免切** | 表格被切成两半 → embedding 搜不到完整表 | 检测 `[TABLE_HTML: Table_N]` 标记跳过 chunker |
| **chunk_size=400** | 学术段落典型长度，多了超 LLM 上下文、少了语义碎片 | 对比 200/400/800 后 400 最平衡 |

### 2.3 向量化与索引层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **BGE-M3** (sentence-transformers) | 中英文双语学术场景，标题英文 + 问题中文混用 | ADA-002 英文 only + 有费用；text2vec 英文弱 |
| **FAISS (IndexFlatIP)** | 10 万级向量精确内积搜索 <10ms | HNSW 更快但有精度损失；IVF 需调参 |
| **rank-bm25 + jieba** | 学术术语精确匹配（"LoRA rank=8"、"batch size 64"） | BM25 对罕见术语的召回远好于向量搜索 |
| **RRF 混合融合 (α=0.6)** | 向量模糊 + BM25 精确 = 两者兼顾 | α 调参实验表明 0.6 时 Recall@10 最高 |
| **独立论文索引** | 每篇论文独立 FAISS + BM25 文件，增删单篇不用重建全库 | 全局索引增删一篇需 30 分钟全量重建 |

### 2.4 检索增强层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **BGE-Reranker** (cross-encoder) | 双编码器 Top-200 粗筛后精排 Top-20 | 比再排序 LLM 便宜（本地跑无需 API） |
| **LLM Listwise Rerank** (DeepSeek) | CrossEncoder 忽略语义上下文时的二次精排 | 对比过 Cohere Rerank（要钱且不如 DeepSeek 准） |
| **QueryRewriter** | 用户口语问题 ↔ 学术术语鸿沟（"最新方法" → "state-of-the-art 2024 survey"） | 单查询召回低，MultiQuery 5 个变体融合提升 2x |
| **意图检测 α 动态调** | 精确查询（"Table 3"、"sensitivity 59.81%"）不要向量泛化 | 固定 α 在所有查询上不是最优 |

### 2.5 LLM 与生成层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **DeepSeek Chat** | 默认 LLM 供应商，性价比高 | GPT-4 贵 10x，Claude 国内访问不便 |
| **引用级标注 Prompt** | 每条回答需要可追溯 → 每句话标注来源 + 文末参考列表 | paper-qa 的核心 Feature，直接对标 |
| **双 Prompt 机制** | 表格检索结果用 TABLE_PROMPT（定性分析），普通块用 QA_PROMPT（引用数值） | 同一个 Prompt 无法兼顾两种模式 |
| **LaTeX 公式输出** | 用户问公式时输出 $$...$$ 数学表达式 | 文字描述公式可读性差 |

### 2.6 对话记忆层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **ContextualQueryRewriter** | "它的准确率？" → 不知道"它"指什么 → 搜不到 | 结合历史用 LLM 重写为独立搜索词 |
| **ConversationMemory** | "那个方法比这个好？" → 不知道哪个方法 | 跨轮追踪论文/方法/指标/表格实体 |
| **滑动窗口 5 轮** | 长对话超出上下文窗口 → 前文丢失 | 不是简单截断，保留结构化摘要 |

### 2.7 存储层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **VectorIOWriter + Python bytes** | FAISS `fopen` 写文件不支持 Windows 中文路径 | `GetShortPathNameW` 依赖 NTFS 8.3 开关，跨平台性差 |
| **FAISS Index + metadata pkl** | 向量索引 + chunk 元数据分离存储 | 元数据灵活更新不影响向量索引 |
| **BM25 pkl 序列化** | BM25 索引状态持久化 | 重建代价高（需全文），必须增量保存 |

### 2.8 Agent 与协议层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **ArxivAgent** (deliberative/reactive) | 用户一句话 → 自主搜索→下载→索引→检索→综述 | 纯 RAG 需要用户手动完成前三步 |
| **MCP Server** | 暴露 4 个 ArXiv 工具给 Claude Desktop 等宿主 | 比 REST API 更标准化（JSON-RPC），比 Function Calling 更通用 |
| **Streamlit 4 Tab** | 搜索/综述/问答/评估 四面板 | 比 Gradio 更 Pythonic，比 Flask+Vue 快 10x 开发速度 |

### 2.9 解决的问题总览

| 类别 | 具体问题 | 技术方案 |
|------|---------|---------|
| 🐛 **PDF 崩溃** | Docling C++ bad_alloc 在 >15 页 PDF 上杀死进程 | 物理拆分为 10 页/批的小 PDF 再分别解析 |
| 🐛 **中文路径** | FAISS write_index 在 `D:\md\AI实习\` 路径上 fopen 失败 | VectorIOWriter bytes 序列化 + Python open() |
| 🐛 **表格丢失** | pdfplumber 网格提取丢失 colspan/rowspan | Docling export_to_html() 原生输出 |
| 🐛 **跨页表格** | Table 1 跨 2-3 页被拆成独立 TableItem | table_map 追踪 + 正则取 `<tr>` 追加合并 |
| 🐛 **切片污染** | 固定窗口切分跨章节 | 章节感知 SectionChunker |
| 🐛 **术语召回差** | 向量搜索把 "LoRA" 模糊成相近词 | BM25 精确匹配 + RRF 混合 α=0.6 |
| 🐛 **代词消解** | "它的准确率？" 搜不到 | ContextualQueryRewriter 结合历史重写 |
| 🐛 **编造内容** | LLM 幻觉无法追溯 | 引用级标注：每句话带 [来源]，文末附参考列表 |
| 🐛 **首次查询慢** | Embedding + Reranker 模型加载 20s+ | Streamlit 启动时预加载 + spinner 提示 |

## 三、基础轮：RAG 与检索

### Q1. 讲讲你的 RAG 整体架构

**简历一句话版：**
> 构建了 PDF 解析→切片→BGE-M3 向量化→FAISS+BM25 混合检索→Rerank→LLM 生成的完整 RAG 流水线

**面试展开版：**

"我的 RAG 架构分五层：

**第一层：解析层。** 一个 try/except 链：先用 Docling（布局树分析，支持表格 HTML），失败降级 pdfplumber（纯文本），再失败 PyPDF2（快速回退）。

**第二层：切片层。** 论文感知切片——先按章节边界分块（Abstract / Introduction / Method / Experiment / Conclusion），每块再按 400 token 子切分，保留 section_title 和 paper_title 元数据。表格块整表落盘不做跨表切。

**第三层：索引层。** BGE-M3（1024 维，支持中英文）向量化 → FAISS IndexFlatIP。同时 BM25（rank_bm25 + jieba 中文分词）做精确关键词通道。每篇论文有独立索引，增删单篇不影响其他。

**第四层：检索层。** 混合检索 RRF 融合（α=0.6 偏 BM25），可选的 BGE-Reranker 交叉编码器重排，加上 LLM Listwise 重排（调用 DeepSeek 给所有候选打分）。

**第五层：生成层。** 引用级标注——LLM 生成的每句话都带 `[论文缩写, 章节名]` 来源标记，末尾列参考来源。这是我实现的最核心的防幻觉机制。"

**面试官可能会追问：**

> **Q：引用级标注具体怎么实现的？**

在 QA_SYSTEM_PROMPT 里加了一条硬性规则："每处事实性信息必须标注来源"。上下文里给 LLM 的每段文本前面有 `[论文缩写]` 前缀，LLM 直接引用这个缩写。末尾要求输出 `参考来源` 列表。纯 Prompt 工程实现，不需要改 pipeline。

> **Q：为什么 α=0.6 偏 BM25？**

学术场景的特殊性。用户搜 "LoRA rank=8" 或 "batch size 64" 时，精确关键词匹配比语义模糊匹配重要得多。

> **Q：BGE-M3 为什么不是别的 Embedding？**

中英文双语能力。论文标题英文、用户问题中文混用场景下，BGE-M3 的 1024 维多语言支持正好匹配。

### Q2. 混合检索是向量好还是 BM25 好？你实际测过吗？

**简历一句话版：**
> 混合搜索 α=0.6 预热后延迟 19ms，优于纯向量 (78ms) 和纯 BM25 (16ms) 的单独效果

**面试展开版：**

"我做了对比测试。10 篇论文、397 个 chunks 上测了 5 个查询：

| 模式 | 平均延迟 | 特点 |
|------|---------|------|
| 纯向量 FAISS | 78ms | 语义匹配好但召回泛化 |
| 纯 BM25 | 16ms | 精确术语命中但语义无关不召回 |
| **混合 α=0.6** | **19ms** | **两者兼顾，默认模式** |

延迟上混合几乎等于 BM25（因为 α 偏 BM25，向量只需粗略算 top 200）。"

### Q3. 表格提取你们怎么做的？

**简历一句话版：**
> 使用 Docling 原生 export_to_html() 保留 colspan/rowspan，跨页表自动合并

**面试展开版：**

"两个阶段：

**1) 格式保留：** 以前用 pdfplumber 的网格提取，合并单元格全丢了。改用 Docling 的 export_to_html() 原生输出，colspan/rowspan 直接保留。

## 四、深挖轮：工程细节

### Q5. Docling OOM 是怎么排查和解决的？

**简历一句话版：**
> 物理拆分 PDF 为每 10 页一组再分别 Docling 解析，完全绕过 C++ bad_alloc

**面试展开版：**

"现象：解析大 PDF（21 页综述论文）时，Docling 的 C++ 预处理层崩溃，进程直接死。

**排查：** 一开始以为是代码 bug。加了 try/except 发现接不住——不是 Python 异常。查进程退出码 `3221225477` = `0xC0000005` = `STATUS_ACCESS_VIOLATION`。看 Docling 源码才发现 preprocess 阶段一次性把所有页面转到位图，大 PDF 直接 `std::bad_alloc`，在 C++ 层触发 SIGABRT，Python 完全没法拦截。

**解决方案：** 物理拆分。用 PyPDF2 把 PDF 按每 10 页拆成独立小文件，分别用 Docling 解析，再合并结果。这样 C++ 层每次只处理 10 页，内存够用。"

---

### Q6. FAISS 中文路径崩溃？

**简历一句话版：**
> 通过 VectorIOWriter 序列化为 bytes 后 Python 读写，绕开 FAISS C++ fopen 对中文路径的限制

**面试展开版：**

"发现是 FAISS 的 C++ `FileIOWriter` 用 `fopen` 写文件。Windows 的 `fopen` 不支持含中文的路径。但 Python 的 `open()` 调用 Windows WideChar API，所以正常工作。

**解决方案：** 用 `faiss.VectorIOWriter` 序列化到内存 bytes → Python `open()` 写。读同理。完全绕过 C++ 文件操作，所有路径处理走 Python 的 Unicode 支持。"

---

### Q7. 三引擎降级链是怎么设计的？

**简历一句话版：**
> Docling → pdfplumber → PyPDF2 三引擎自动降级

**面试展开版：**

"设计原则：**先到先得，自动降级。** 三个引擎按优先级排列：
1. **Docling（优先）** — 布局树分析，支持表格 HTML
2. **pdfplumber（降级）** — 纯文本 + 表格网格提取
3. **PyPDF2（兜底）** — 纯文本快速提取

这个设计的好处：部署在不同机器上自动适配，不用改一行代码。"

**简历一句话版：**

**面试展开版：**

### Q9. 你的切片策略为什么是按章节切？和固定窗口比？

**简历一句话版：**

## 五、系统设计轮

### Q10. 如果让你扩展到 10 万篇论文，架构怎么改？

**简历一句话版：**
> 单体 FAISS → Milvus 分布式向量库 + Celery 异步解析流水线 + Collection 分片

**面试展开版：**

"1. **向量存储：** 单机 FAISS → Milvus / Qdrant。FAISS 单机 10 万 × 1024 维没问题（约 400MB），但并发和容错不够。
2. **解析流水线：** 单进程 → Celery + RabbitMQ 任务队列。PDF 下载 / Docling 解析 / Embedding 三步异步编排。
3. **检索分片：** 按 ArXiv 分类（cs.CV, cs.CL, cs.LG）做 collection 分片。
4. **缓存：** Redis 缓存高频查询结果。"

### Q11. 并发 1000 QPS 时瓶颈在哪？

**简历一句话版：**
> 解析阶段 CPU 密集是瓶颈，检索阶段 Reranker 是瓶颈

**面试展开版：**

"写路径（解析）：Docling CPU 密集。一篇 21 页论文 ~150s 单核 CPU。
读路径（检索）：FAISS（78ms）和 BM25（16ms）不是瓶颈。瓶颈在 Reranker：BGE-Reranker CrossEncoder 每对 ~50ms。方案：退化到仅 LLM Rerank。"

## 六、Behavioral 轮

### Q12. 你做这个项目的动机？

"读论文的痛点。研究生每天要：上 ArXiv 搜论文、下载 PDF、自己读、自己对比方法、手写综述。我想做一个一站式的工具——搜索→下载→自动解析→直接问答和综述生成。"

### Q13. 一个人做全栈，最大的收获？

"学会了做减法。RAG 可以做得非常复杂——文档排序、知识图谱、多跳检索。但一个项目要在有限时间内可交付，我必须判断什么值得做深、什么做到够用就行。"

### Q14. 你觉得自己代码写得怎么样？

"核心链路的代码质量不错——有类型注解、异常处理、降级策略、每篇论文独立索引。但测试覆盖率不足是短板。"
