## 二、技术栈全景：用了什么，解决什么问题

> 面试官最爱问「你为什么选这个技术而不是那个」  
> 每一行都要能回答「解决了什么痛点」

### 2.1 PDF 解析层

| 技术                          | 解决的核心问题                                                   | 为什么不是别的                                         |
| --------------------------- | --------------------------------------------------------- | ----------------------------------------------- |
| **Docling**                 | 布局树分析：区分标题/段落/表格，原生 `export_to_html()` 保留 colspan/rowspan | pyMuPDF 没布局树；pdfplumber 网格提取丢合并单元格；Camelot 安装复杂 |
| **pdfplumber**              | Docling 降级回退——没 GPU 或 Docling 崩溃时纯文本提取                    | 比 PyPDF2 准（有文字定位），比 Docling 轻                   |
| **PyPDF2**                  | 最后兜底——最轻量纯文本提取                                            | 准度差（无定位），只保证有东西                                 |
| **Docling 分批 (page_range)** | 大 PDF 的 C++ `std::bad_alloc` OOM 崩溃，Python 接不住            | 事前检测页数拆批，不走 try/except 失败重试                     |

### 2.2 切片层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **SectionChunker** | 固定窗口切分把 Method 尾和 Experiment 首切进同一块 → 上下文污染 | 按章节边界切（Abstract/Intro/Method/Exp/Conclusion），每块语义完整 |
| **表格原子块免切** | 表格被切成两半 → embedding 搜不到完整表 | 检测 `[TABLE_HTML: Table_N]` 标记跳过 chunker |
| **chunk_size=400** | 学术段落典型长度，多了超 LLM 上下文、少了语义碎片 | 对比 200/400/800 后 400 最平衡 |

### 2.3 向量化与索引层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **BGE-M3** (sentence-transformers) | 中英文双语学术场景，标题英文 + 问题中文混用 | ADA-002 英文 only + 有费用；text2vec 英文弱 |
| **FAISS (IndexFlatIP)** | 10 万级向量精确内积搜索 <10ms | HNSW 更快但有精度损失；IVF 需调参 |
| **rank-bm25 + jieba** | 学术术语精确匹配（"LoRA rank=8"、"batch size 64"） | BM25 对罕见术语的召回远好于向量搜索 |
| **RRF 混合融合 (α=0.6)** | 向量模糊 + BM25 精确 = 两者兼顾 | α 调参实验表明 0.6 时 Recall@10 最高 |

### 2.4 检索增强层

| 技术                                 | 解决的核心问题                                                  | 为什么不是别的                             |
| ---------------------------------- | -------------------------------------------------------- | ----------------------------------- |
| **BGE-Reranker** (cross-encoder)   | 双编码器 Top-200 粗筛后精排 Top-20                                | 比再排序 LLM 便宜（本地跑无需 API）              |
| **LLM Listwise Rerank** (DeepSeek) | CrossEncoder 忽略语义上下文时的二次精排                               | 对比过 Cohere Rerank（要钱且不如 DeepSeek 准） |
| **QueryRewriter**                  | 用户口语问题 ↔ 学术术语鸿沟（"最新方法" → "state-of-the-art 2024 survey"） | 单查询召回低，MultiQuery 5 个变体融合提升 2x      |
| **意图检测 α 动态调**                     | 精确查询（"Table 3"、"sensitivity 59.81%"）不要向量泛化               | 固定 α 在所有查询上不是最优                     |

### 2.5 LLM 与生成层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **DeepSeek Chat** | 默认 LLM 供应商，性价比高 | GPT-4 贵 10x，Claude 国内访问不便 |
| **双 Prompt 机制** | 表格检索结果用 TABLE_PROMPT（定性分析），普通块用 QA_PROMPT（引用数值） | 同一个 Prompt 无法兼顾两种模式 |
| **表格 HTML 后处理** | LLM 看不到 HTML 表格的数值 → 提示词要求提取 + 规整格式 | 纯文本表格丢失结构信息 |

### 2.6 存储层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **VectorIOWriter + Python bytes** | FAISS `fopen` 写文件不支持 Windows 中文路径 | `GetShortPathNameW` 依赖 NTFS 8.3 开关，跨平台性差 |
| **FAISS Index + metadata pkl** | 向量索引 + chunk 元数据分离存储 | 元数据灵活更新不影响向量索引 |
| **BM25 pkl 序列化** | BM25 索引状态持久化 | 重建代价高（需全文），必须增量保存 |

### 2.7 Agent 与协议层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **ArxivAgent** (deliberative/reactive) | 用户一句话 → 自主搜索→下载→索引→检索→综述 | 纯 RAG 需要用户手动完成前三步 |
| **MCP Server** | 暴露 4 个 ArXiv 工具给 Claude Desktop 等宿主 | 比 REST API 更标准化（JSON-RPC），比 Function Calling 更通用 |
| **Streamlit 4 Tab** | 搜索/综述/问答/评估 四面板 | 比 Gradio 更 Pythonic，比 Flask+Vue 快 10x 开发速度 |

### 2.8 配置与评估层

| 技术 | 解决的核心问题 | 为什么不是别的 |
|------|-------------|--------------|
| **Pydantic Config** (AppConfig 聚合) | 9 个子配置统一管理 + `.env` 文件覆盖 | `yaml` 无类型校验；`json` 不支持继承 |
| **Langfuse** | 检索阶段延迟 / token 用量 / 用户查询追踪 | 开源可自建，vs LangSmith 要付费 |
| **EvaluationTracker** | Recall@K / MRR / NDCG 离线指标计算 | 自己写一套轻量，不需要集成评估框架 |

### 2.9 解决的问题总览

| 类别 | 具体问题 | 技术方案 |
|------|---------|---------|
| 🐛 **PDF 崩溃** | Docling C++ bad_alloc 在 >15 页 PDF 上杀死进程 | pdfplumber 快速取页数 → 拆 10 页/批 |
| 🐛 **中文路径** | FAISS write_index 在 `D:\md\AI实习\` 路径上 fopen 失败 | VectorIOWriter bytes 序列化 + Python open() |
| 🐛 **表格丢失** | pdfplumber 网格提取丢失 colspan/rowspan | Docling export_to_html() 原生输出 |
| 🐛 **跨页表格** | Table 1 跨 2-3 页被拆成独立 TableItem | table_map 追踪 + 正则取 `<tr>` 追加合并 |
| 🐛 **Embedding 搜不到表** | 仅 HTML tag 作为搜索文本 | 平铺为 caption + 列名 + 前2行数值 |
| 🐛 **切片污染** | 固定窗口切分跨章节 | 章节感知 SectionChunker |
| 🐛 **术语召回差** | 向量搜索把 "LoRA" 模糊成相近词 | BM25 精确匹配 + RRF 混合 α=0.6 |
| 🐛 **首次查询慢** | Embedding + Reranker 模型加载 20s+ | Streamlit 启动时预加载 + spinner 提示 |

## 三、基础轮：RAG 与检索

### Q1. 讲讲你的 RAG 整体架构

**简历一句话版：**
> 构建了 PDF 解析→切片→BGE-M3 向量化→FAISS+BM25 混合检索→Rerank→LLM 生成的完整 RAG 流水线

**面试展开版：**

"我的 RAG 架构分五层：

**第一层：解析层。** 一个 try/except 链：先用 Docling（布局树分析，支持表格 HTML），失败降级 pdfplumber（纯文本），再失败 PyPDF2（快速回退）。Docling 仅对 TableItem 生成 HTML `<table>`，其他内容保持 Markdown 文本。

**第二层：切片层。** 论文感知切片——先按章节边界分块（Abstract / Introduction / Method / Experiment / Conclusion），每块再按 400 token 子切分，保留 section_title 和 paper_title 元数据。表格块整表落盘不做跨表切。

**第三层：索引层。** BGE-M3（1024 维，支持中英文）向量化 → FAISS IndexFlatIP。同时 BM25（rank_bm25 + jieba 中文分词）做精确关键词通道。

**第四层：检索层。** 混合检索 RRF 融合（α=0.6 偏 BM25），可选的 BGE-Reranker 交叉编码器重排，加上 LLM Listwise 重排（调用 DeepSeek 给所有候选打分）。

**第五层：生成层。** 双 Prompt 机制——检索到表格块时用 TABLE_SYSTEM_PROMPT（定性分析，禁止搬数字），否则用 QA_SYSTEM_PROMPT。"

**面试官可能会追问：**

> **Q：为什么 α=0.6 偏 BM25？**

因为学术场景的特殊性。用户搜 "LoRA rank=8" 或 "batch size 64" 时，精确关键词匹配比语义模糊匹配重要得多。BM25 擅长精确术语匹配，向量搜索会过度泛化。我在 15 篇论文上做了手动调参，α=0.6 时 Recall@10 最高。

> **Q：BGE-M3 为什么不是别的 Embedding？**

中英文双语能力。论文标题英文、用户问题中文混用场景下，BGE-M3 的 1024 维多语言支持正好匹配。对比 ADA-002（英文 only、有成本）、text2vec（英文弱）。

> **Q：首次查询很慢吧？怎么解决的？**

是的。首次查询涉及 Embedding 模型 + Reranker 加载，约 20s。解决方案：在 Streamlit 启动时预加载，`st.spinner` 显示进度。后续查询 Embedding 编码仅 14ms。

### Q2. 混合检索是向量好还是 BM25 好？你实际测过吗？

**简历一句话版：**
> 混合搜索 α=0.6 预热后延迟 19ms，优于纯向量 (78ms) 和纯 BM25 (16ms) 的单独效果

**面试展开版：**

"我做了对比测试。4 篇论文、124 个 chunks 上测了 5 个查询：

| 模式 | 平均延迟 | 特点 |
|------|---------|------|
| 纯向量 FAISS | 78ms | 语义匹配好但召回泛化 |
| 纯 BM25 | 16ms | 精确术语命中但语义无关不召回 |
| **混合 α=0.6** | **19ms** | **两者兼顾，默认模式** |

延迟上混合几乎等于 BM25（因为 α 偏 BM25，向量只需粗略算 top 200）。质量上 BM25 保证精确术语召回到，向量提供同义扩展。实际用户体验：搜 'sensitivity' 也能召回含 'recall' 的论文段落。"

**面试官可能会追问：**

> **Q：α 是固定的吗？**

不，我实现了意图检测动态调 α。检测到精确查询（Table 引用、数字、方法名）时 α→0.8 偏 BM25；模糊问题（"最新进展"）时 α→0.4 偏向量。代码在 `_detect_intent()` 里。

### Q3. 表格提取你们怎么做的？

**简历一句话版：**
> 使用 Docling 原生 export_to_html() 保留 colspan/rowspan，跨页表自动合并，表格内容平铺为 Embedding 文本

**面试展开版：**

"这是项目里工程最复杂的一块。三个阶段：

**1) 格式保留：** 以前用 pdfplumber 的网格提取，合并单元格全丢了——colspan=2 变成空 + 实两个 `<th>`，表格完全不可读。改用 Docling 的 export_to_html() 原生输出，colspan/rowspan 直接保留。

**2) 跨页合并：** 学术论文的大表格经常跨 2-3 页，Docling 每页输出独立 TableItem。我做了 `table_map` 字典追踪每个 Table_N，检测到相同 table_id 出现在不同页时——或者 caption 含 `(continued)` 标记时——用正则提取新块的 `<tr>` 追加到已有表块的 `</table>` 前。

**3) Embedding 可检索：** 表格的向量文本不是直接取 HTML，而是平铺为 'caption | 列名1 列名2 列名3 | 行1值1 值2 值3 | row_header 值1 值2' 的格式。这样用户搜 '59.81% accuracy NBI' 时，向量能命中含这些数值的表格。"

**面试官可能会追问：**

> **Q：跨页表合并有没有误合并的情况？**

有。如果两页都有 `Table_1` 但指不同表格（概率极低，论文里极少出现），会误合并。所以加了 `last_page` ≠ `current_page` 的防误判约束——同一页内出现两个 Table_1 不会合并。

### Q4. Query 改写怎么做？

**简历一句话版：**
> 支持 Academic / MultiQuery / HyDE / Step-Back / chinese_academic 五种改写策略

**面试展开版：**

"我实现了 5 种策略，按查询意图自动选择：

1. **Academic**：口语→术语，'前沿目标检测' → 'state-of-the-art object detection survey 2024'
2. **MultiQuery**：生成 5 个子查询各自检索后融合，提高召回
3. **HyDE**：先假设回答再以假回答做检索——有些论文不直接包含搜索词但包含回答
4. **Step-Back**：先问一个更抽象的问题获取上下文，再细查
5. **chinese_academic**：检测到中文时自动触发，中文问题切换为英文搜索词

高频查询实际效果提升：不加改写时 'D-Caps 和 Iv3 对比' 第一篇命中不到 40%，加 MultiQuery 后提到 80%。"

## 四、深挖轮：工程细节

### Q5. Docling OOM 是怎么排查和解决的？

**简历一句话版：**
> 自动检测 PDF 页数，超过 15 页拆为 10 页/批处理，保留完整表格提取能力

**面试展开版 （准备的最深 bug 故事之一）：**

"现象：解析大 PDF（21 页综述论文）时，Docling 的 C++ 预处理层崩溃，进程直接死。

**排查过程：**

一开始以为是代码 bug。加了 try/except 发现接不住——不是 Python 异常。查进程退出码 `3221225477` = `0xC0000005` = `STATUS_ACCESS_VIOLATION`。看 Docling 源码才发现 preprocess 阶段一次性把所有页面转到位图，大 PDF 直接 `std::bad_alloc`，在 C++ 层触发 SIGABRT，Python 完全没法拦截。

**解决方案：**

事前防御：先用 pdfplumber 快速获取页数（3ms），不超过 15 页直接全文 Docling，超过 15 页拆成 10 页/批，每批调用 `converter.convert(page_range=(start, end))`。批次间通过 `page_range` 参数保证页码绝对值正确——`item.prov[0].page_no` 是原始 PDF 页号，不是批次内偏移。这样就做到了：PDF 不管多大都保留 Docling 的表格提取能力，不会 OOM。"

### Q6. FAISS 中文路径崩溃？

**简历一句话版：**
> 通过 VectorIOWriter 序列化为 bytes 后 Python 读写，绕开 FAISS C++ fopen 对中文路径的限制

**面试展开版：**

"现象：项目在 `D:\md\AI实习\` 下，FAISS 的 `write_index()` 写入 `faiss_papers.index` 时报 `could not open ... for writing`。但路径确定存在，Python 文件操作正常。

**排查：**

发现是 FAISS 的 C++ `FileIOWriter` 用 `fopen` 写文件。Windows 的 `fopen` 不支持含中文的路径（使用 ANSI 编码，非 UTF-8）。但 Python 的 `open()` 调用 Windows WideChar API，所以正常工作。

**解决方案：**

用 `faiss.VectorIOWriter` 序列化到内存 bytes → `bytes(faiss.vector_to_array(writer.data))` → Python `open()` 写。读同理。完整读写链路：

```
# 写
writer = faiss.VectorIOWriter()
faiss.write_index(index, writer)
idx_bytes = bytes(faiss.vector_to_array(writer.data))
with open(path, 'wb') as f: f.write(idx_bytes)

# 读
with open(path, 'rb') as f: data = f.read()
vec = faiss.UInt8Vector()
faiss.copy_array_to_vector(np.frombuffer(data, dtype=np.uint8), vec)
reader = faiss.VectorIOReader(); reader.data = vec
index = faiss.read_index(reader)
```

完全绕过 C++ 文件操作，所有路径处理走 Python 的 Unicode 支持。"

**面试官可能会追问：**

> **Q：为什么不用 GetShortPathNameW？**

试过。`GetShortPathNameW` 依赖 NTFS 的 8.3 短名功能，但 Windows 10/11 默认关闭，需要管理员 `fsutil 8dot3name set` 开启。VectorIOWriter 是跨平台通用方案，Linux/Mac 同样工作，不需要依赖 Windows 特定功能。

---

### Q7. 三引擎降级链是怎么设计的？

**简历一句话版：**
> Docling → pdfplumber → PyPDF2 三引擎自动降级，使用文件锁 + 信号量保证并发安全

**面试展开版：**

"设计原则：**先到先得，自动降级。** 三个引擎按优先级排列：

1. **Docling（优先）** — 布局树分析，支持表格 HTML。CPU 密集、内存占用高
2. **pdfplumber（降级）** — 纯文本 + 表格网格提取，轻量但丢 colspan/rowspan
3. **PyPDF2（兜底）** — 纯文本快速提取，最轻量

每个引擎用独立的 lock 控制并发，进程启动时清空目录锁避免死锁。只要一个引擎返回非空文本就算成功。

这个设计的好处：部署在不同机器上（无 GPU 的服务器自动用 pdfplumber，有 GPU 的用 Docling），不用改一行代码。"

---

### Q8. 你的切片策略为什么是按章节切？和固定窗口比？

**简历一句话版：**
> 论文感知章节切片，按 Abstract/Introduction/Method/Experiment/Conclusion 边界切分

**面试展开版：**

"固定窗口切（比如 400 token 滑动）会把 Method 最后一句话和 Experiment 第一句话切到同一个 chunk，造成上下文污染。按章节切有两个优点：

1. **语义完整**：一个 chunk 只含一个章节的内容，检索时 LLM 看到的信息不跨话题
2. **元数据丰富**：每个 chunk 带 section_title，文件名路由可以做 section-level 术语暴击加分

学术论文有高度结构化的特点——Introduction 和 Experiment 讨论的问题完全不同。利用这个结构是最好的切片策略。

表格块是例外——整表免切，通过 `[TABLE_HTML: Table_X]` 标记跳过 chunker，整表作为一个原子块落盘。"

## 五、系统设计轮

### Q9. 如果让你扩展到 10 万篇论文，架构怎么改？

**简历一句话版：**
> 单体 FAISS → Milvus 分布式向量库 + Celery 异步解析流水线 + Collection 分片

**面试展开版：**

"我会做四个改动：

**1) 向量存储：** 单机 FAISS → 分布式向量数据库（Milvus / Qdrant / Pinecone）。FAISS 单机支撑 10 万 × 1024 维没问题（约 400MB），但并发和容错不够。Milvus 的 IVF_FLAT 索引在 10 万量级搜索 <10ms，支持多副本。

**2) 解析流水线：** 单进程 → Celery + RabbitMQ 任务队列。PDF 下载 / Docling 解析 / Embedding 编入索引三步异步编排，每步失败自动重试。

**3) 检索分片：** 按 ArXiv 分类（cs.CV, cs.CL, cs.LG）做 collection 分片。用户查询自动路由到对应分类的 collection 搜索。

**4) 缓存：** Redis 缓存高频查询结果。Query 改写做 cache key 归一化，'最新目标检测' 和 'object detection latest' 映射到同一 key。"

---

### Q10. 并发 1000 QPS 时瓶颈在哪？

**简历一句话版：**
> 解析阶段 CPU 密集是瓶颈，检索阶段 Reranker 是瓶颈

**面试展开版：**

"分读写路径分析：

**写路径（解析）：** Docling CPU 密集。一篇 21 页论文 ~147s 单核 CPU。瓶颈在解析。方案：GPU 加速 + 预转换，新论文提交后后台异步解析。

**读路径（检索）：** 向量搜索（78ms）和 BM25（16ms）不是瓶颈——纯内存操作。瓶颈在 **Reranker**：BGE-Reranker CrossEncoder 每对 ~50ms，top-20 候选 × 50ms = 1s。方案：退化到仅 LLM Rerank（一次 DeepSeek API 调用排序全部候选），或者把 Reranker 部署在 GPU 上。

**LLM 生成：** API 调用 2-5s 是硬延迟。方案：流式 SSE 返回、Redis 缓存高频问题、Fallback 到缓存摘要。"

## 六、Behavioral 轮

### Q11. 你做这个项目的动机？

"读论文的痛点。研究生每天要：上 ArXiv 搜论文、下载 PDF、自己读、自己对比方法、手写综述。我想做一个一站式的工具——搜索→下载→自动解析→直接问答和综述生成。

技术上，这个项目刚好覆盖了 RAG 全链路 + Agent 自主规划 + MCP Server，是一个能完整展示工程能力的作品。"

### Q12. 一个人做全栈，最大的收获？

"学会了做减法。RAG 可以做得非常复杂——文档排序、知识图谱、多跳检索。但一个项目要在有限时间内可交付，我必须判断什么值得做深、什么做到够用就行。

好的例子：跨页表合并做了（用户看到完整表格，体验差别大）。坏的例子：多语言翻译支持没做（项目才 15 篇英文论文，做了浪费）。这种判断力是只有自己从 0 到 1 做才能练出来的。"

### Q13. 你觉得自己代码写得怎么样？

"诚实说：核心链路的代码质量不错——有类型注解、异常处理、降级策略。但测试不足是短板。下一个迭代我会优先补 pytest，保证每个新功能不会破坏旧功能。"
