# ArXiv Scholar — AI Agent 开发实习面试题优化版

> 目标：把这个项目讲成一个“能落地的 RAG + Agent + MCP 系统”，而不是只会堆 LangChain / Prompt 的 Demo。  
> 使用方式：先背“0. 快速话术”和“1. 核心数字”，再按 Q&A 深挖。面试时回答不要一次讲太满，先给结构，再等面试官追问。

## 0. 面试开场必背

### 0.1 30 秒项目定位

我做的 ArXiv Scholar 是一个面向科研读论文场景的智能学术助手，核心链路是 **ArXiv 搜索 → PDF 下载 → 多引擎解析 → 章节感知切片 → BGE-M3 向量化 → FAISS + BM25 混合检索 → Reranker 精排 → LLM 生成引用级回答**。  
在这个基础上，我又做了 Agent 自主工作流和 MCP Server，让模型可以自己完成“改写 Query、搜索论文、下载 PDF、增量建索引、检索问答、生成综述”这一套流程。

### 0.2 1 分钟项目介绍

这个项目解决的是研究生读论文时的真实痛点：查论文、下载 PDF、读方法、对比实验、写综述需要在多个工具之间来回切换。我希望把它做成一个端到端系统。  
我主要做了三层：

1. **RAG 检索层**：PDF 解析默认用 pdfplumber（本地秒级），同时支持 Docling / MinerU 多引擎兜底；切片不是固定窗口，而是按论文章节组织；检索使用 BGE-M3 + FAISS 做语义召回，同时用 BM25 保住学术术语和模型名，再用 BGE-Reranker 或 LLM listwise 精排。
2. **Agent 工作流层**：Agent 根据问题意图自动做 Query 改写，比如 MultiQuery、HyDE、Step-Back、中文学术查询改写，然后调用搜索、下载、索引、RAG 检索、综述生成等工具。
3. **工程稳定性层**：我处理过 Docling C++ 层 OOM、FAISS 中文路径、跨页表格合并、引用级溯源、多轮代词消解等问题，所以这个项目不只是 Demo，而是比较完整的大模型应用工程实践。

### 0.3 3 分钟深度版

如果面试官说“展开讲讲”，按下面顺序讲：

1. **为什么做**：读论文链路太碎，ArXiv 数据公开，适合做真实 AI Agent 应用。
2. **系统怎么跑**：自然语言问题先做意图识别和 Query 改写；如果需要新论文，先调 ArXiv API 搜索并下载；PDF 进入解析、切片、向量化和 BM25 索引；查询时走混合检索 + rerank；最后 LLM 基于上下文生成带来源的答案。
3. **技术选择**：BGE-M3 适合中英双语，BM25 对模型名和指标更稳，章节切片比固定窗口更符合论文结构，MCP 让工具可被外部 Agent 标准调用。
4. **最难的坑**：Docling OOM 不是 Python 异常，是 C++ 层进程崩溃，所以 try/except 接不住；最后用 PyPDF2 物理拆 PDF 为 10 页一组再分别解析。FAISS 中文路径是 C 层 fopen 不支持 Unicode，我用 VectorIOWriter 序列化到 bytes，再交给 Python open 写入。
5. **结果与反思**：项目能支撑本地论文库问答、引用溯源和自动综述；下一步如果要上生产，需要把 FAISS 换成 Milvus / Qdrant，把 PDF 解析改成异步任务队列，并引入 Redis 缓存和更系统的评测集。

## 1. 面试前必须脱口而出的数字

| 维度 | 数字 / 参数 | 面试时怎么说 |
|---|---:|---|
| Chunk 大小 | `chunk_size=512` | 学术段落长，512 tokens 能保留较完整语义 |
| Chunk 重叠 | `overlap=128` | 约 25% 重叠，避免关键句被切断 |
| Embedding | BGE-M3 / 1024 维 | 支持中英文，适合中文问英文论文 |
| 向量索引 | FAISS `IndexFlatIP` | 小中规模精确检索，先保证效果和可解释性 |
| 检索候选 | vector top30 + BM25 top30 → hybrid top20 → rerank top10 | 先召回，再精排，避免一开始就让 reranker 扫全库 |
| 混合权重 | 当前代码公式：`score = alpha * vector + (1-alpha) * BM25`，默认 `alpha=0.6` | 注意：代码里 alpha 是向量权重；如果强调 BM25，应改成 0.4 或调整注释 |
| 解析兜底 | pdfplumber（默认）/ Docling / MinerU / PyPDF2 | 从轻量文本解析到布局解析逐级可选；默认 pdfplumber 保证首次建库体验 |
| Docling OOM 方案 | 每 10 页物理拆批 | C++ 层崩溃 try/except 接不住，只能降低单次处理规模 |
| 本地验证数据 | ~10 篇论文、300+ chunks | 实际跑通问答、综述和 RAGAS 评估 |
| 评估指标 | RAGAS 三指标 + 检索层指标 | Faithfulness / Answer Relevancy / Context Precision + Recall@K / MRR / NDCG |
| 评估 LLM | 与问答共用 LLM（默认 deepseek-chat），`max_tokens=1024` | 评估需足够 token 输出逐条分析 + JSON |

## 2. 项目概览类问题

### Q1. 你这个项目一句话怎么介绍？

**推荐回答：**  
这是一个面向科研论文场景的 RAG + Agent + MCP 系统。它不仅能做本地论文问答，还能让 Agent 自主完成 ArXiv 搜索、PDF 下载、论文解析、增量建索引、混合检索和综述生成。相比普通 PDF 问答 Demo，我重点做了论文结构感知切片、向量 + BM25 混合检索、引用级溯源、MCP 工具封装和工程异常兜底。

**面试官追问点：** 为什么不是简单地把 PDF 塞进向量库？  
**回答要点：** 论文有明显结构，方法、实验、结论的语义边界很重要；学术术语也很精确，纯向量容易漏掉模型名、指标名和数据集名；所以我做了章节感知切片 + BM25 精确召回 + rerank。

### Q2. 你为什么选择 ArXiv 论文场景？

**推荐回答：**  
第一，ArXiv 数据公开、API 稳定，适合做可复现的项目。第二，研究生读论文是高频真实需求，不是为了套技术而找场景。第三，这个场景天然覆盖 RAG、Agent、工具调用、PDF 解析、引用追踪、评估等大模型应用开发核心能力，所以很适合作为 AI Agent 开发实习岗位的作品。

### Q3. 这个项目里你最核心的贡献是什么？

**推荐回答：**

1. 独立搭建了 PDF 到引用级回答的端到端 RAG 链路。
2. 把检索从“单纯向量搜索”升级为章节切片、向量 + BM25 混合召回、rerank 精排和多轮上下文改写。
3. 做了 Agent 自主工作流和 MCP Server，把搜索、下载、检索、综述这些能力工具化。
4. 解决了多个工程问题，比如 Docling C++ OOM、FAISS 中文路径、跨页表格和引用溯源。

### Q4. 如果面试官质疑“这是不是课程项目 / Demo”？

**推荐回答：**  
这个项目确实是个人项目，但我不是只调一个库做演示。我做了完整的数据流和异常处理：PDF 解析有多引擎兜底，索引能增量构建，检索有向量和 BM25 双路融合，回答有引用级溯源，Agent 工具有 MCP 封装，评估有 Langfuse 和检索指标。更重要的是，我实际遇到并解决了底层工程问题，例如 Docling C++ 崩溃和 FAISS Unicode 路径问题，这些通常不是简单 Demo 会遇到的。

## 3. RAG 架构与检索深挖

### Q5. 讲讲你的 RAG 整体架构。

**推荐回答：**

我的 RAG 分为离线建库和在线查询两条链路。

**离线建库：**

1. ArXiv API 搜索并下载 PDF。
2. PDF 解析：默认 pdfplumber（本地秒级），可选 Docling / MinerU 获取布局和表格，失败后回退 PyPDF2。
3. 章节感知切片：按论文结构识别 Abstract、Introduction、Method、Experiment、Conclusion 等章节，再在章节内部做 512 token、128 overlap 的切片。
4. Embedding：使用 BGE-M3 生成 1024 维向量。
5. 建索引：FAISS 存向量，BM25 存分词后的文本，同时保存 source、section_title、section_id、page 等 metadata。

**在线查询：**

1. 对用户问题做上下文改写和学术 Query 改写。
2. 并行走 FAISS 向量召回和 BM25 关键词召回。
3. 对两路分数归一化后融合。
4. 用 BGE-Reranker 或 LLM rerank 精排。
5. 把 top chunks 和来源元数据交给 LLM，生成带引用的回答。

### Q6. 为什么要做混合检索？纯向量不行吗？

**推荐回答：**  
纯向量适合语义相近的问题，但学术场景有很多精确词，比如 LoRA、D-Caps、F1-score、batch size、数据集名字，这些词在 BM25 里更容易被精确命中。纯 BM25 又不擅长语义改写，比如“这个方法有什么创新”和“method contribution”语义相关但字面不同。  
所以我用 FAISS 做语义召回，用 BM25 保住精确术语，再融合后 rerank。这样能同时兼顾语义泛化和关键词精确匹配。

**追问：融合公式是什么？**  
当前实现是先把 vector_score 和 bm25_score 分别 min-max 归一化，再计算：

```text
score = alpha * vector_score + (1 - alpha) * bm25_score
```

这里要注意，代码里的 `alpha` 是向量权重，默认 0.6 表示向量略高。如果我要强调“学术场景 BM25 更重要”，更严谨的做法是把 alpha 调成 0.4，或者把变量命名改成 `vector_weight`，避免注释和公式含义不一致。

### Q7. 为什么用 BGE-M3？

**推荐回答：**  
因为这个项目有典型的跨语言检索：用户可能用中文问，论文大部分是英文。BGE-M3 对中英文语义对齐比较好，输出 1024 维向量，适合做中英混合语义召回。相比只面向英文的 embedding，BGE-M3 在中文 Query 搜英文 Chunk 时更稳。

### Q8. 为什么还需要 Reranker？

**推荐回答：**  
Embedding 检索是双塔结构，查询和文档分别编码，速度快但交互不充分。Reranker 是交叉编码器，会把 query 和 candidate 拼在一起判断相关性，效果更好但更慢。所以我先用 FAISS + BM25 召回 top candidates，再只对小候选集做 rerank。这样成本可控，排序质量也更好。

### Q9. 你的切片策略为什么不是固定窗口？

**推荐回答：**  
固定窗口容易把一个方法描述或实验结论切断，也可能把 Method 和 Experiment 混在一起。论文天然有章节结构，所以我先识别章节，再在章节内部切片，并保存 section_title、section_id 等 metadata。这样有三个好处：

1. 检索结果能知道来自哪个章节，便于引用溯源。
2. 面对“这篇论文的方法是什么”这类问题，可以优先召回 Method 相关 chunk。
3. 对长列表或连续实验结果，可以做 sibling chunk 扩展，把同一 section 的相邻块一起带上，避免答案只拿到半截。

### Q10. 章节识别失败怎么办？

**推荐回答：**  
章节识别不是 100% 准，所以我做了 fallback。如果无法识别章节，就退回到段落切片。面试时我会主动说明这是一个工程取舍：优先利用论文结构，但不能因为结构识别失败导致整个建库失败，所以需要兜底策略。

### Q11. 中文 Query 怎么搜英文论文？

**推荐回答：**  
我做了两层处理。第一层是 embedding 选型，用 BGE-M3 做中英文语义对齐。第二层是 Query 改写，中文问题会先改写成更学术化的英文或中英混合搜索词，比如把“胶囊网络检测息肉效果怎么样”改成 “capsule network polyp detection accuracy sensitivity specificity”。这样向量检索和 BM25 都更容易命中。

### Q12. Query 改写有哪些策略？

**推荐回答：**

1. **Academic Rewrite**：把口语问题改成学术搜索词。
2. **MultiQuery**：生成多个不同角度的查询，提高召回。
3. **HyDE**：先让模型生成一个假想答案或摘要，再用它去检索。
4. **Step-Back**：把具体问题抽象成更宽泛的问题，适合查背景和方法综述。
5. **chinese_academic**：中文问题先识别术语，再转成更适合英文论文检索的表达。

**追问：Query 改写会不会引入噪声？**  
会，所以我不会无限扩展 Query。工程上限制改写数量，并在融合阶段让多 Query 的召回结果一起竞争，最后交给 reranker 精排。如果改写失败，就回退原始 Query。

### Q13. 你怎么降低 RAG 幻觉？

**推荐回答：**

1. **检索层**：混合检索 + rerank 提高上下文相关性。
2. **Prompt 层**：要求模型只基于给定上下文回答，不确定就说无法从当前论文判断。
3. **引用层**：每个关键结论绑定来源 `[论文缩写, 章节名]`，最后给出参考来源列表。
4. **交互层**：多轮对话中做代词消解，避免“它的准确率”这类问题丢失指代对象。

### Q14. 引用级标注是怎么做的？

**推荐回答：**  
每个 chunk 在入库时都会保存 source、section_title、page、chunk_id 等 metadata。检索时把这些 metadata 和 chunk text 一起传给 LLM，Prompt 要求模型在回答句子后附上 `[论文缩写, 章节名]`。最后再把本轮使用到的 sources 聚合成参考来源列表。  
这个设计不能从数学上完全保证没有幻觉，但它能显著提高可追溯性：用户看到结论后能回到对应论文和章节核查。

## 4. Agent 与 MCP 深挖

### Q15. 你的 Agent 做了什么？和普通 RAG 有什么区别？

**推荐回答：**  
普通 RAG 是用户问什么就检索什么；Agent 多了任务规划和工具调用。比如用户说“帮我找最近关于 diffusion model medical image segmentation 的论文并总结方法”，Agent 需要先改写 Query，再搜索 ArXiv，选择论文，下载 PDF，解析建索引，最后 RAG 检索并生成综述。  
所以我的 Agent 不是只做聊天，而是把搜索、下载、索引、检索、综述这些动作串成一个自主工作流。

### Q16. 你实现了哪些工具？

**推荐回答：**

1. `search_papers`：调用 ArXiv API 搜论文。
2. `download_paper`：下载 PDF 到本地并登记 metadata。
3. `rag_query`：对本地论文库做混合检索和问答。
4. `rewrite_query`：把自然语言改写成学术查询。
5. MCP Server 里还封装了 `get_paper_info`、`list_local_papers` 等工具，方便外部 Agent 调用。

### Q17. 为什么要做 MCP？

**推荐回答：**  
MCP 的价值是把工具能力标准化。以前每个 Agent 框架都要自己写工具适配层；用了 MCP 后，我的 ArXiv 搜索、下载、本地论文列表等能力可以通过统一协议暴露给外部模型或 Agent 客户端。这样工具和模型解耦，后续换模型、换 Agent 框架，工具层不用大改。

### Q18. MCP 和 Function Calling 有什么区别？

**推荐回答：**  
Function Calling 更像模型厂商提供的“单次工具调用格式”，核心是让模型输出结构化参数，然后由应用执行函数。MCP 更像一个工具协议和运行时约定，定义了工具发现、参数 schema、调用方式和服务边界。  
我的理解是：Function Calling 解决“模型怎么调用函数”，MCP 解决“工具怎么被不同模型和 Agent 客户端标准化发现和调用”。

### Q19. Agent 容易失控，你怎么保证稳定性？

**推荐回答：**

1. 工具参数用 schema 约束，避免模型乱传参数。
2. 工具执行失败要返回结构化错误，而不是直接崩溃。
3. 对搜索数量、下载数量、重试次数设置上限。
4. 对关键步骤保留日志和 trace，方便复盘。
5. 对需要高确定性的任务，优先用固定 pipeline；Agent 只负责规划和调度，不让它随意改底层检索逻辑。

## 5. 工程深挖：STAR 故事

### Q20. 讲一个你遇到的最深 Bug：Docling OOM。

**S 背景：**  
我用 Docling 解析 PDF，因为它能保留布局树和表格结构。但解析较大的论文时，进程直接崩溃。

**T 任务：**  
要保证 PDF 解析稳定，不能因为单篇论文大一点就让整个建库流程挂掉。

**A 行动：**  
一开始我用 try/except 包住解析，发现接不住；后来排查到这是 C++ 层预处理阶段的内存问题，进程级崩溃不是普通 Python Exception。于是我改成物理拆批：用 PyPDF2 把 PDF 每 10 页拆成一个小 PDF，再分别交给 Docling 解析，最后合并文本和表格结果。

**R 结果：**  
把不可捕获的 C++ 崩溃转成可控的小批次解析问题，显著提高了解析稳定性。这个故事体现的是：遇到第三方库底层崩溃时，不能只在 Python 层加 try/except，要改变输入规模或隔离进程。

### Q21. FAISS 中文路径问题怎么解决？

**推荐回答：**  
在 Windows 上，项目路径里有中文时，FAISS 的 C++ `fopen` 写索引可能失败。Python 的 `open()` 没问题，因为它走 Windows Unicode API，但 FAISS C 层不一定支持。  
我用了两种思路：第一是在配置里尽量拿 Windows 短路径；第二是在向量库保存时用 `faiss.VectorIOWriter` 把索引序列化到内存 bytes，再用 Python `open()` 写入文件。读取时也反过来用 bytes 交给 FAISS。这样路径处理交给 Python，绕过 C 层 Unicode 限制。

### Q22. 跨页表格怎么处理？

**推荐回答：**  
PDF 里的表格经常跨页，被解析器切成多个表。我用 table metadata 做追踪，如果后续页的表格结构和前一页连续，就把 HTML 表格行追加到同一个逻辑表里，而不是当成两个无关表。这个问题的本质是 PDF 没有天然的逻辑结构，解析器只能给局部结果，应用层要根据页码、表头和表格形态做合并。

### Q23. 为什么做多引擎 PDF 解析？

**推荐回答：**  
因为 PDF 解析没有银弹。MinerU / Docling 对布局和表格更好，但依赖更重、速度更慢，也可能失败；pdfplumber 更轻，适合文本和部分表格；PyPDF2 最轻，适合作为兜底。我的策略是优先保证高质量解析，失败时自动降级，保证系统可用性。

### Q24. 你怎么处理增量建索引？

**推荐回答：**  
下载新论文后，不应该每次全量重建。我给论文维护 registry 和 metadata，新增论文解析后只对新 chunks 做 embedding，再 append 到 FAISS 和 BM25，同时保存 per-paper index 信息。这样用户新增论文后能快速入库，避免重复解析历史 PDF。

### Q25. 如果解析结果质量不高怎么办？

**推荐回答：**  
我会分层处理：首先在解析层保留更丰富的结构，比如 Docling 的布局树和表格 HTML；其次在切片层过滤过短或明显无意义的块；再在检索层通过 rerank 把噪声 chunk 降权；最后在生成层要求模型基于引用回答。  
但我也会诚实说明：PDF 解析质量是 RAG 系统上限之一，尤其是复杂公式、多栏、扫描件和跨页表格，需要更强的 VLM 解析或人工校验。

## 6. 系统设计与扩展问题

### Q26. 如果扩展到 10 万篇论文，你怎么改架构？

**推荐回答：**

1. **向量库**：FAISS 单机可以支撑 10 万级向量，但生产上要考虑并发、更新和容灾，所以会切到 Milvus / Qdrant，并按 ArXiv category 或用户空间做 collection / partition。
2. **解析流水线**：PDF 下载、解析、embedding、入库拆成异步任务，用 Celery / RabbitMQ 或云队列，避免用户请求阻塞在解析上。
3. **缓存**：Redis 缓存高频 Query、论文 metadata、搜索结果和 Query 改写结果。
4. **检索优化**：先按论文类别、时间、文件名或用户选择做过滤，再做向量 / BM25 召回；大规模下引入 ANN 索引，比如 HNSW 或 IVF。
5. **评估与监控**：构建固定评测集，监控 Recall@K、MRR、NDCG、Latency、生成引用命中率和用户反馈。

### Q27. 如果并发 1000 QPS，瓶颈在哪里？

**推荐回答：**  
读路径和写路径要分开看。

**写路径瓶颈** 是 PDF 解析和 embedding，尤其 Docling / MinerU 解析很重，应该异步化并横向扩容。  
**读路径瓶颈** 主要可能在 reranker 和 LLM 生成，FAISS / BM25 本身相对快。优化方式是：缓存高频查询、减少 rerank 候选数、使用轻量 reranker、批量 embedding、流式输出，并对 LLM 做限流和降级。

### Q28. 为什么现在用 FAISS，而不是一开始就 Milvus？

**推荐回答：**  
项目早期目标是验证检索效果和完整链路，不是先上分布式组件。FAISS 部署简单、无服务依赖、调试方便，适合个人项目快速迭代。等数据量、并发和多用户隔离成为主要问题，再切 Milvus / Qdrant 更合理。这个取舍体现的是先把核心闭环跑通，再做架构升级。

### Q29. 如果让你上线给真实用户用，还缺什么？

**推荐回答：**

1. 用户体系和多租户数据隔离。
2. 异步任务队列和任务状态查询。
3. 更完整的评测集和线上反馈闭环。
4. 成本控制：embedding、rerank、LLM token、缓存策略。
5. 安全：文件上传校验、工具调用权限、Prompt Injection 防护。
6. 更好的前端交互：引用跳转、PDF 高亮、综述导出。

### Q30. 如何评估这个 RAG 系统效果？

**推荐回答：**  
我会分检索和生成两层评估。

**检索层** 用 Recall@K、MRR、NDCG、Latency，评估相关 chunk 是否被召回、排得是否靠前、延迟是否可接受。  
**生成层** 用 correctness、faithfulness、citation accuracy、answer relevance 来评估答案是否正确、是否基于上下文、引用是否对应。  
项目里已经有 Langfuse trace 和自定义检索指标，后续要补的是固定 golden set，比如每篇论文准备 5-10 个标准问题和标准证据 chunk。

## 7. 高频追问与防守回答

### Q31. 你项目最大短板是什么？

**推荐回答：**  
最大短板是还没有大规模线上用户和系统化标注评测集。现在主要是本地论文库和自构造问题验证，能证明链路可行，但还不足以证明真实生产环境下的长期效果。我的下一步会补 golden set、用户反馈、线上监控和更完整的 A/B 对比。

### Q32. 你觉得最有技术含量的地方是什么？

**推荐回答：**  
不是单独某个模型调用，而是把论文 RAG 做成一条稳定链路：PDF 解析质量、章节切片、混合检索、rerank、引用溯源、多轮改写、Agent 工具调用，每一环都会影响最终答案。尤其是 Docling OOM 和 FAISS 中文路径这些问题，让我更理解大模型应用开发不只是 Prompt，而是数据工程、检索工程和系统工程的组合。

### Q33. 如果不用 LangChain / LangGraph，你还能做吗？

**推荐回答：**  
可以。这个项目的核心不是框架，而是状态流和工具抽象。不用 LangGraph，也可以自己定义状态对象，包括 query、plan、tool_results、retrieved_chunks、final_answer，然后用普通 Python 函数串起来。LangGraph 的价值是让状态流、节点、边和失败重试更清晰，适合复杂 Agent 工作流，但不是不可替代。

### Q34. 你的项目和普通 ChatPDF 有什么区别？

**推荐回答：**

1. 普通 ChatPDF 多数是单 PDF 问答，我的是 ArXiv 搜索 + 多论文本地库。
2. 普通 ChatPDF 往往是固定切片 + 向量检索，我做了章节感知切片、BM25 混合检索和 rerank。
3. 我做了引用级溯源、多轮代词消解和 Agent 自主下载建库。
4. 我把工具封装为 MCP Server，可以被外部 Agent 标准调用。

### Q35. 你怎么解释“Agent 开发能力”？

**推荐回答：**  
我理解的 Agent 开发不是让模型自由聊天，而是把任务拆成可控的状态流和工具调用链。它包括工具 schema 设计、参数校验、执行结果结构化、错误重试、trace 观测、任务上限控制，以及在合适的地方保留确定性 pipeline。我的 ArXiv Scholar 就是把论文检索、下载、解析、索引、问答、综述这些能力封装成工具，让 Agent 做规划和调度。

### Q36. 如果面试官指出 alpha 注释和公式不一致怎么办？

**推荐回答：**  
我会承认这是一个需要修正的地方。当前 `_fuse_scores` 里的公式是 `alpha * vector + (1-alpha) * bm25`，所以 alpha 是向量权重。配置注释写“学术场景 BM25 更重要”不够严谨。如果要保持这个公式，我应该把默认 alpha 调到 0.4；或者保留 0.6，但注释改成“语义向量略高，同时保留 BM25 权重”。这个问题也说明我会看实现，而不是只背 README。

### Q37. 如果问你“有没有真实效果数据”？

**推荐回答：**  
我现在有本地验证数据，比如 10 篇论文、329 个 chunks、34 张表格，能跑通问答、引用和综述链路；也记录了候选召回和延迟指标。但我会诚实区分“工程验证”和“严格实验评测”：严格评测还需要 golden set 和人工标注相关 chunk，这是我下一步要补的。

## 8. Behavioral 轮 STAR 话术

### Q38. 你为什么做这个项目？

**推荐回答：**  
读研后我发现读论文是高频任务，但流程很碎：搜索、下载、读方法、对比实验、写综述都在不同工具里。我想做一个真正能帮自己提高效率的工具，同时这个场景也能系统练到 RAG、Agent、MCP 和评估。最后我把它做成了一个从 ArXiv 搜索到本地论文问答的完整链路。

### Q39. 一个人做全栈最大的收获是什么？

**推荐回答：**  
最大的收获是学会做取舍。RAG 能做得很复杂，比如知识图谱、多跳检索、多模态解析、复杂评测，但个人项目必须先保证核心闭环。我先把 PDF 解析、切片、检索、生成、引用这条主链路跑通，再逐步加 Agent、MCP、评估和异常兜底。这个过程让我更理解工程落地不是功能越多越好，而是关键路径要稳定。

### Q40. 你遇到困难时怎么推进？

**推荐回答：**  
我会先判断问题属于哪一层：模型问题、检索问题、解析问题还是工程环境问题。比如 Docling OOM，表面看是解析失败，但本质是 C++ 层进程崩溃；FAISS 中文路径，表面看是保存失败，本质是 C 层文件 API 不支持 Unicode。定位清楚层次后，再选方案：降级、拆批、绕过底层 API 或换组件。

### Q41. 你代码写得怎么样？

**推荐回答：**  
我不会说自己代码已经很完美，但我比较重视模块化和可维护性。项目里把 arxiv_client、parser、chunker、embedding、retriever、agent、mcp、evaluation 分成独立模块；配置集中在 config；运行数据固定在 data 目录，避免依赖启动目录。后续如果继续优化，我会补更多单元测试、类型检查和 CI。

## 10. 最后冲刺清单

### 10.1 必须背熟的 8 个故事

1. 项目 30 秒定位：RAG + Agent + MCP 的学术论文助手。
2. RAG 全链路：解析 → 切片 → embedding → FAISS + BM25 → rerank → 引用级回答。
3. 混合检索：纯向量 vs BM25 vs rerank 的取舍。
4. Query 改写：Academic / MultiQuery / HyDE / Step-Back / chinese_academic。
5. Docling OOM：C++ 层崩溃，物理拆批解决。
6. FAISS 中文路径：VectorIOWriter bytes 序列化绕过 C 层 fopen。
7. 引用级溯源：metadata + prompt 约束 + 来源列表。
8. 扩展到生产：Milvus / Qdrant + Celery + Redis + golden set。

### 10.2 面试中不要踩的坑

1. 不要说“完全解决幻觉”，要说“降低幻觉并提高可追溯性”。
2. 不要把 MCP 说成模型本身能力，它是工具协议。
3. 不要说 FAISS 一定比 Milvus 好，要说项目阶段不同，选型不同。
4. 不要说 alpha=0.6 偏 BM25；当前代码公式里 alpha 是向量权重。
5. 不要只讲框架名，要讲数据流、状态流、异常处理和评估闭环。

### 10.3 最像 AI Agent 实习生的表达

> 我理解 Agent 开发的核心不是让模型自由发挥，而是把任务拆成可控状态流，把外部能力封装成稳定工具，再通过评估和 trace 不断收敛效果。我的项目正好覆盖了工具 schema、RAG 检索、MCP 封装、工作流编排、异常兜底和评估闭环，所以我希望在实习中继续做更真实的 Agent 落地项目。
