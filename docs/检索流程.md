# ArXiv Scholar 检索流程

## 整体流程

```
用户提问
    │
    ├─→ [1] 上下文改写 ──→ ContextualQueryRewriter（代词消解）
    │
    ├─→ [2] 双通道并行检索
    │     │
    │     ├─→ RAG 文本检索（pipeline.query）
    │     │     ├── Query 改写（MultiQuery / Academic / HyDE）
    │     │     ├── FAISS 向量搜索（BGE-M3 编码）
    │     │     ├── BM25 关键词搜索
    │     │     ├── RRF 混合融合（α=0.6）
    │     │     ├── BGE-Reranker 重排
    │     │     └── LLM Listwise 重排
    │     │           │
    │     │           ▼
    │     │     Top-K 文本 chunks
    │     │
    │     └─→ 图床搜索（figure_extractor）
    │           ├── 检测 "Figure X" / "Table X"
    │           ├── search_by_caption("Figure 16")
    │           ├── 过滤到当前论文
    │           ├── 展开 ±3 页（覆盖跨页）
    │           └── 取该页所有图 + 整页截图
    │                 │
    │                 ▼
    │           图片 blobs（SQLite）
    │
    ├─→ [3] 上下文组装
    │     ├── 文本 chunks → ctx 字符串
    │     ├── 图表命中 → 注入 【原图可用】标记
    │     ├── 对话历史 → history_messages
    │     └── 结构化记忆 → memory_context
    │
    ├─→ [4] LLM 生成回答
    │     ├── QA_SYSTEM_PROMPT / TABLE_SYSTEM_PROMPT
    │     ├── 含【原图可用】标记让 LLM 知道有图
    │     └── 回答 + 图注交叉验证（防幻觉）
    │
    └─→ [5] 前端展示
          ├── LLM 回答（Markdown + LaTeX）
          ├── 图床图片（最多 5 张，优先 figure/table）
          ├── 表格原图（SQLite）或 HTML 降级
          └── 保存到 chat_messages + 更新记忆
```

## 关键设计点

### 为什么文本和图表是分开检索的？

因为文本存在 FAISS + BM25 索引中，图表存在 SQLite 图床中。两者是独立的存储系统：

| | 文本 | 图表 |
|--|------|------|
| **存什么** | 切片后的文本 chunks | PNG 图片二进制 |
| **存在哪** | FAISS + BM25 + metadata.pkl | SQLite (figures.db) |
| **怎么搜** | 向量语义搜索 + 关键词 | SQL LIKE caption 搜索 |
| **返回什么** | 文本片段 + 元数据 | 图片 bytes + caption |

### 为什么图表不是 RAG 搜出来的？

因为图片不能向量化（我们没有多模态模型）。所以单独用 SQLite 做 caption 精确匹配。用户问 "Figure 16" 时走 SQL LIKE，问 "PR曲线" 时走文本 RAG 找到相关论文，再关联到该论文的图表。

### 为什么 LLM 还是会幻觉？

LLM 的上下文只包含 `【原图可用】table_1: Table 1. Summary of Original Datasets` 这样的文字标记，**它看不到图片本身**。所以数据的准确性依赖：
1. Docling 从 PDF 提取的文本质量（表格数据常错位）
2. LLM 自己的先验知识（数据集表格的标准格式）
3. 用户自己看原图验证（最终保障）
